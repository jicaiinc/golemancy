# 需求清单：深度 Code Review（交叉验证）
> 创建时间：2026-02-16 15:00
> 状态：已确认

## 功能需求
1. 对项目代码进行非常深入的代码审查，深度追踪代码逻辑，不是浅层扫描
2. Team Lead 二次校验：三位 CR 角色产出审查报告后，Team Lead 亲自阅读并二次校验，判断报告中的问题是否真实、准确、合理
3. 团队交叉验证：CR 角色之间的结论需要交叉验证，由其他角色或事实验证师确认问题是否是**事实**，而非误判或过度解读
4. Review 结论必须基于事实（官方文档、源码证据、最新最佳实践），不能基于猜测

## 审查范围（6 大重点领域）
5. **Sandbox 功能**：审查 sandbox 的功能设置，配置是否合理、实现是否正确
6. **Permission 系统**：审查 permission mode（restricted sandbox / unrestricted sandbox）的安全性，同时从**业务逻辑和功能可用性**角度审查——权限模式是否好用、逻辑是否合理、用户体验是否顺畅
7. **Code Runtime 功能**：审查打包进去的 Python 环境和 Node.js 环境是否好用、是否正确工作，对代码运行时的逻辑进行深度追踪
8. **编译打包流程**：审查构建（build）和打包（package）的过程，配置是否正确、流程是否可靠
9. **测试相关功能**：审查测试体系（单元测试、集成测试、E2E 测试）的覆盖度和质量
10. **整体代码库**：`packages/ui/`、`packages/server/`、`packages/shared/`、`apps/desktop/` 全范围审查

## 审查视角
11. **安全性视角**：不仅是传统安全（XSS/注入），还要从**功能安全性**角度审查 permission 和 sandbox 的业务逻辑
12. **可用性视角**：从实际使用角度判断功能是否好用、逻辑是否通顺
13. **代码逻辑深度追踪**：对关键功能（特别是 code runtime、sandbox、permission）进行端到端的代码逻辑追踪

## 流程要求
14. CR-Quality / CR-Security / CR-Performance 三者并行进行深度审查
15. 产出审查报告后，Team Lead 对每份报告进行二次校验
16. 对于有争议或重要的发现，事实验证师进行事实交叉验证
17. 最终产出经过交叉验证的、可信的审查报告

## 注意事项
18. "深度"意味着要追踪完整的代码执行路径，不能只看表面
19. Permission 审查要兼顾安全性和可用性两个维度
20. Code Runtime 要验证打包的 Python/Node.js 环境实际是否可用
21. 交叉验证是核心——任何结论都必须有第二方确认
