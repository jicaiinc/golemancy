## 一、用户痛点分析

### 痛点 1：Leader 不可删除 → 整个 Team 报废

**根本原因**：数据模型中 `TeamMember.parentAgentId === undefined` 即为 leader，代码 `useTeamTopologyData.ts:217-218` 中 `removeMember` 显式拒绝删除无 `parentAgentId` 的成员。UI 层面 `TeamNodeDetailPanel.tsx:165` 也隐藏了 leader 的删除按钮。

**影响**：
- 第一个添加的 agent 自动成为 leader，之后无法更换
- 用户想重新排布拓扑时，leader 成为不可移除的钉子
- 只能删除整个 team 重建，工作量大且丢失 instruction 等配置
- 用户原话："我又想把这个节点删除重新排布，但是我删除了 sub agent，对于这个 team leader agent 我就没法删除"

**解决方向**：允许删除任意节点（包括 leader），或提供 leader 切换/重新指定机制。删除 leader 时需要处理其子节点的归属问题（自动提升、重新挂载、或变为孤立节点）。

---

### 痛点 2：新增节点重叠

**根本原因**：`TeamTopologyView.tsx:115-117` 中添加子节点时，位置计算仅考虑父节点的正下方偏移（`parentNode.position.y + height + 60`），**完全不考虑已有兄弟节点的位置**。虽然 `useTeamTopologyLayout.ts` 使用 dagre 做自动布局，但 dagre 只在 `needsLayout` 为 true（存在无 savedPosition 的节点）时运行，且新节点被预先存入 `savedLayoutRef`，导致 dagre 认为该节点已有位置而跳过布局计算。

**影响**：
- 添加第二个子节点时，直接覆盖在第一个子节点上方
- 用户必须手动拖拽分开，体验极差
- 用户原话："第二个 sub agent 它会和第一个 sub agent 重叠"

**解决方向**：添加节点后应自动触发智能布局（考虑兄弟节点位置），或使用增量式布局算法，避免所有新节点都塞在同一个位置。

---

### 痛点 3：Team Instruction 交互别扭

**根本原因**：`TeamTopologyToolbar.tsx:100-102` 用工具栏按钮 toggle 一个绝对定位面板（`TeamTopologyToolbar.tsx:152-163`），面板从工具栏底部展开。无动画过渡，blur 时自动保存但无明确反馈，视觉上像是临时弹窗而非正式配置区域。

**影响**：
- 作为重要配置项，交互方式过于随意
- 没有"保存"的明确操作，用户不确定修改是否生效
- 点开/收起的体验像是临时弹窗，不像正式功能
- 用户原话："这种配置都让人感觉的非常的别扭，这在设计层面上是很垃圾的"

**解决方向**：将 Team Instruction 整合到一个持久化的配置面板中（如侧边栏或独立的 Settings 区域），而非工具栏的临时展开。

---

### 痛点 4：整体 UI 又丑又难看又不好用

**根本原因（综合分析）**：
1. **工具栏过于拥挤** — 名称编辑、描述编辑、Instruction 按钮、Reset Layout、Delete Team 全挤在一行，且描述行用了 invisible spacer hack 对齐（`TeamTopologyToolbar.tsx:130-131`）
2. **详情面板覆盖画布** — 右侧 300px 面板直接叠在 ReactFlow 上方（absolute），遮挡拓扑图
3. **空状态简陋** — 仅一个文字 + 按钮，无视觉引导
4. **Agent 选择器原始** — 添加节点时使用简单下拉列表，无搜索/过滤/分组
5. **节点设计信息过载** — 节点内展示 skills、tools、MCP、memory 全部信息，导致高度不一致且密集
6. **缺少拓扑全局视角** — 没有清晰的层级视觉（level indicator, depth coloring 等）
7. **用户原话**："当前的实现我觉得又丑又难看又不好用，然后 UI 布局也很奇怪"

**解决方向**：完全从零重设计，不参考当前实现。

## 二、功能清单与优先级

### P0 — 核心功能（必须实现，解决用户明确痛点）

| # | 功能 | 对应痛点 | 说明 |
|---|------|---------|------|
| P0-1 | Leader 机制灵活化 | 痛点 1 | 允许删除 leader（子节点处理策略）+ 允许切换 leader（右键/拖拽提升） |
| P0-2 | 智能自动布局 | 痛点 2 | 添加节点后自动计算不重叠的位置；支持一键 re-layout |
| P0-3 | Team Instruction 交互重设计 | 痛点 3 | 从临时 toggle 改为正式配置区域 |
| P0-4 | 整体视觉重设计 | 痛点 4 | 全新布局、节点样式、工具栏设计，符合 pixel art 风格 |

### P1 — 重要改进（提升体验，建议在重设计中一并实现）

| # | 功能 | 说明 |
|---|------|------|
| P1-1 | 节点设计简化与优化 | 节点卡片只展示核心信息（名称、角色、状态），详细信息移至详情面板 |
| P1-2 | 详情面板不遮挡画布 | 面板独立于画布区域（右侧固定栏或 split view），不用 absolute 覆盖 |
| P1-3 | Agent 选择器增强 | 添加 agent 时提供搜索、过滤、分组功能；或改为拖拽添加 |
| P1-4 | 拓扑层级视觉增强 | 不同层级用颜色/缩进/连线样式区分，让层级关系一目了然 |
| P1-5 | 空状态引导优化 | 提供有意义的空状态插画或步骤引导 |
| P1-6 | 工具栏布局优化 | Team 元信息（名称、描述）和操作按钮合理分区 |

### P2 — 锦上添花（可在后续迭代实现）

| # | 功能 | 说明 |
|---|------|------|
| P2-1 | 拖拽重组父子关系 | 拖拽节点到另一个节点上方时自动重新挂载 |
| P2-2 | 批量操作 | 多选节点后批量删除/移动/设置父节点 |
| P2-3 | 快捷键支持 | Delete 删除选中节点、Ctrl+A 全选、Ctrl+Z 撤销 |
| P2-4 | 布局动画 | 布局变化时节点平滑过渡动画 |

## 三、验收标准（可测试）

### P0-1：Leader 机制灵活化

- [ ] **AC-1.1** 任何节点（包括 leader）都可以被删除
- [ ] **AC-1.2** 删除 leader 时：
  - 若 leader 有子节点，子节点自动变为根级节点（各自成为独立的 leader 或保持无 parent 状态）
  - 用户可以看到子节点状态变化的提示
- [ ] **AC-1.3** 支持通过明确交互（如右键菜单或工具栏操作）将任意节点设为新 leader
- [ ] **AC-1.4** 团队允许存在 0 个或多个无 parent 的根节点（不强制 single-root 约束）
- [ ] **AC-1.5** 删除最后一个成员后，回到空状态，team 不会报错

### P0-2：智能自动布局

- [ ] **AC-2.1** 添加新节点后，新节点与所有现有节点不重叠（间距 ≥ 40px）
- [ ] **AC-2.2** 有一键 re-layout 功能，重新计算所有节点位置（tree layout）
- [ ] **AC-2.3** 手动拖拽后的位置被记忆（刷新页面后保持）
- [ ] **AC-2.4** 新节点添加位置考虑兄弟节点：同一父节点下的子节点水平/垂直均匀分布

### P0-3：Team Instruction 交互重设计

- [ ] **AC-3.1** Team Instruction 有专属的、固定的编辑区域（非临时弹窗/toggle）
- [ ] **AC-3.2** 编辑后有明确的保存状态指示（如 "已保存" 提示或自动保存 + 状态标记）
- [ ] **AC-3.3** 编辑区域在视觉上与 Team 配置（名称、描述）同属一个配置面板
- [ ] **AC-3.4** Instruction 输入框支持多行文本，高度可调或自适应

### P0-4：整体视觉重设计

- [ ] **AC-4.1** 遵循 pixel art / Minecraft 暗色风格（无圆角、像素字体、shadow-pixel 系列阴影）
- [ ] **AC-4.2** 页面布局清晰分区：工具栏/元信息区、画布区、详情/配置区
- [ ] **AC-4.3** 视觉层次分明：leader vs sub-agent 有明确视觉区分
- [ ] **AC-4.4** 整体视觉质量明显优于当前实现（由 UI/UX 设计师主导，非简单修补）

### P1-1：节点设计简化

- [ ] **AC-5.1** 节点卡片只展示：agent 名称、角色标签、状态指示灯、model 名称
- [ ] **AC-5.2** Skills、Tools、MCP、Memory 等详细信息仅在选中节点后的详情面板中展示
- [ ] **AC-5.3** 所有节点高度一致或接近一致（误差 ≤ 20px），避免布局跳动

### P1-2：详情面板不遮挡画布

- [ ] **AC-6.1** 详情面板打开时，画布区域自动收缩（flex 布局或 split），而非被遮挡
- [ ] **AC-6.2** 面板有打开/关闭的过渡动画
- [ ] **AC-6.3** 面板中包含：agent 完整信息、角色编辑、capabilities 概览、操作按钮

### P1-3：Agent 选择器增强

- [ ] **AC-7.1** 选择器支持按名称搜索过滤
- [ ] **AC-7.2** 已在 team 中的 agent 不出现在选择器中（当前已实现，需保持）
- [ ] **AC-7.3** 选择器中展示 agent 的模型和简要描述

### P1-4：拓扑层级视觉增强

- [ ] **AC-8.1** Leader 节点有明显的视觉强调（如更大尺寸、特殊边框/图标）
- [ ] **AC-8.2** 父→子连线样式清晰，方向明确（如箭头/渐变色）
- [ ] **AC-8.3** 不同层级的节点有视觉区分（可选：颜色渐变、缩进、标记等）

### P1-5：空状态引导

- [ ] **AC-9.1** 空 team 展示像素风格的引导视觉（非纯文字）
- [ ] **AC-9.2** 提供明确的第一步操作入口

### P1-6：工具栏布局优化

- [ ] **AC-10.1** Team 元信息（名称、描述）与操作按钮（reset layout、delete 等）视觉分离

## 四、技术约束与注意事项

1. **数据模型兼容性**：`TeamMember.parentAgentId === undefined` 表示 leader 的语义可能需要调整。如果允许多个无 parent 节点，需确认 server 端 `agent/sub-agent.ts` 对 leader 的依赖逻辑。
2. **布局库**：当前使用 `@dagrejs/dagre`，可评估是否需要更换（如 ELK、d3-hierarchy），由 Architect / Fact Checker 决定。
3. **ReactFlow 版本**：当前使用 `@xyflow/react`（ReactFlow v12），重设计需继续使用此库。
4. **i18n**：所有新增 UI 文本需使用 `t()` 函数，namespace 为 `team`。
5. **样式**：遵循 Tailwind CSS v4 + pixel art 风格，无圆角，使用项目已有的 design token。
6. **性能**：节点数量预计在 2-20 之间，无需特殊虚拟化处理。
