Team Topology: DAG vs Tree 对比

场景：游戏开发公司——一款大型多人在线游戏的开发团队

▸ DAG 结构（共享角色）
同一个 Agent 被多个 Parent 直接调用。紫色虚线节点 = 共享角色，紫色虚线边 = 共享调用关系。
观察：紫色虚线就是"共享"带来的视觉复杂度。QA Agent 被 5 条紫线连接，L10n Agent 被 3 条。
线条交叉严重，尤其是 QA Agent 的连线横跨整个画面从最左的 Concept Lead 到右边的 Level Lead。
真实场景中，如果再多几个共享角色（如 DevOps Agent、Security Agent），画面会更混乱。
▸ Tree 结构（复制替代共享）
每个需要 QA/L10n 的 Lead 有自己的专属实例。配置相同，但互相独立。绿色节点 = 复制的实例。
对比：所有连线都是垂直/近垂直向下，无交叉，无横跨。
代价：QA 有 4 个副本、L10n 有 3 个副本。但运行时行为和 DAG 完全相同。
未来可以通过 Agent Template 功能统一管理配置更新。
▸ 对比总结
  • DAG：QA Agent 有 5 条紫色虚线从不同方向汇入，线条交叉横跨整个画布
  • DAG：L10n Agent 有 3 条紫色虚线，与 QA 的线混在一起难以分辨
  • DAG：如果再加 Security Agent、DevOps Agent 等共享角色，复杂度指数增长
  • DAG：用户很难通过视觉追踪某条线从哪来到哪去
  • Tree：所有连线方向一致（上→下），无交叉
  • Tree：层级结构一目了然，每个节点的 parent 清晰可见
  • Tree：副本数量增加，但可通过 Template 机制管理
  • Tree：拖拽编辑时不会影响其他不相关的线
