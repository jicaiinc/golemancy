Multi-Team Analysis: 同一批 Agent 的多种编排

场景：AI 软件开发工作室 —— 8 个 Agent 组成 5 个不同的 Team 来处理不同类型的任务

① Agent Pool（项目中的所有 Agent）
每个 Agent 是独立的能力单元：有自己的 System Prompt、工具、记忆。
记忆属于 Agent 本身——无论在哪个 Team 中工作，它带着同样的知识和经验。
Architect 系统架构师 🧠 记忆: 项目架构决策、技术债记录 🔧 Bash, 文件读写
Frontend 前端开发 🧠 记忆: 组件库规范、样式约定 🔧 Bash, Browser, 文件读写
Backend 后端开发 🧠 记忆: API设计规范、数据模型 🔧 Bash, 文件读写, DB
DBA 数据库专家 🧠 记忆: Schema变更历史、索引策略 🔧 Bash, DB, 文件读写
QA 测试工程师 🧠 记忆: 已知bug模式、测试覆盖盲区 🔧 Bash, Browser, 文件读写
DevOps 运维工程师 🧠 记忆: 部署配置、故障处理手册 🔧 Bash, 文件读写
Security 安全工程师 🧠 记忆: 漏洞修复记录、安全审计标准 🔧 Bash, 文件读写
TechWriter 技术文档 🧠 记忆: 文档风格指南、术语表 🔧 文件读写
关键点：Agent 的记忆是持久且私有的
Architect 记得所有架构决策历史，QA 记得所有已知 bug 模式——
这些知识在不同 Team 中都能发挥作用。这是"同一个 Agent 参与多个 Team"的核心价值。
如果复制 Agent，记忆就割裂了。
② 5 个 Team 编排（同一批 Agent，不同层级结构）
每个 Team 处理一种任务类型。紫色边框 = 该 Agent 同时出现在其他 Team 中。

Team 1: Feature Development

新功能开发，最常用的团队配置

Team 2: Bug Triage

线上问题排查——QA 主导，开发配合调查

Team 3: Security Audit

安全审计——Security 主导，审查架构和代码

Team 4: Release Pipeline

版本发布——DevOps 主导，QA 验证，文档同步

Team 5: Performance Optimization

性能优化——Architect 主导，全栈 + DBA + DevOps 协作排查和优化

关键发现
8 个 Agent 中有 6 个参与了 3+ 个 Team。 同一个 Agent 在不同 Team 中角色不同：

· Architect：Team 1/5 中是 Leader（架构主导），Team 3 中是 Member（被 Security 审查）
· QA：Team 2 中是 Leader（bug 排查主导），Team 1/4 中是 Member（配合开发/发布）
· Backend：参与 4 个 Team，每次带着同样的 API 设计记忆，但执行不同任务

记忆的跨 Team 价值：Backend 在 Team 1 开发新 API 时积累的知识， 在 Team 2 排查 bug 时直接可用，在 Team 3 安全审查时也能提供上下文。 如果为每个 Team 复制独立 Backend Agent，这些知识就割裂了。

分析结论

多 Team 是真实且高频的需求，原因有三：

1. 任务类型多样性：同一个项目有开发、测试、发布、审计等不同工作流， 每种工作流需要不同的 Leader 和不同的层级结构。这不是边缘场景——而是日常。

2. Agent 记忆的跨 Team 价值：Agent 的持久记忆是它的核心资产。 Backend Agent 在开发中积累的 API 知识，在 bug 排查和安全审计中同样有价值。 复制 Agent 会割裂记忆，违背了"Agent = 有经验的个体"的设计理念。

3. 角色灵活性：同一个 Agent 在不同 Team 中扮演不同角色 （Architect 有时是 Leader，有时是被审查的 Member）。 这种灵活性是 Agent 编排的核心价值——不是 Agent 变了，是组织结构变了。

建议：多 Team 应该是一等公民（first-class）功能，不是边缘场景。 数据模型必须原生支持同一个 Agent 参与多个 Team。
⑥ 对数据模型的影响
好消息：当前数据模型已经支持

Team.memberAgentIds: AgentId[] 天然允许同一个 AgentId 出现在多个 Team 中。
Agent 不持有 Team 引用——Team 单向引用 Agent。这意味着：

· 创建新 Team 时，从 Agent Pool 中选择成员即可
· 同一个 Agent 可以被任意数量的 Team 引用
· 删除 Team 不影响 Agent
· Agent 的记忆与 Team 无关，始终跟随 Agent 本身

不需要改数据模型——只需要在 UI 层清晰地展示"这个 Agent 还参与了哪些其他 Team"。
