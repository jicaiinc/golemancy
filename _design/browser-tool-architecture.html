核心设计原则
Tool 定义层只写一次，永远不需要改。 新增 Driver（插件/云端/移动端）时，只需要实现 BrowserDriver 接口， 上层 8 个 tool 定义 + Agent 调用链路零改动。

共享 vs 分别实现
从上面两个流程可以看到：

Agent → Tool 定义 → BrowserDriver 接口 这三层是 完全共享 的，代码只写一次。

BrowserDriver 接口 → 底层执行 这层是 分别实现 的，每种模式有自己的 Driver。

所以浏览器工具不需要写两遍。8 个 tool 定义在 tools.ts 中写一次，通过 BrowserDriver 接口调用。 只有 Driver 层面需要分别实现 PlaywrightDriver 和 ExtensionDriver。

三、共享 vs 分别实现 · 逐层详解

层 共享/分别 文件位置 说明
Agent Runtime 共享 ✓ packages/server/src/agent/runtime.ts 已有代码，streamText + tools，无需改动
Tool Loading 共享 ✓ packages/server/src/agent/builtin-tools.ts 已有代码，加一个 if (config.browser) 分支
Tool Definitions 共享 ✓ packages/tools/src/browser/tools.ts 8 个 tool 定义 (browser_navigate, browser_click...) — 只写一次
Snapshot 序列化 共享 ✓ packages/tools/src/browser/snapshot.ts PageSnapshot → 文本格式化 + ref 生成算法 — 两种模式输出格式一致
BrowserDriver 接口 共享 ✓ packages/tools/src/browser/driver.ts TypeScript interface — 契约定义
PlaywrightDriver 分别 ✗ packages/tools/src/browser/drivers/playwright.ts playwright-core 实现 BrowserDriver
ExtensionDriver 分别 ✗ packages/tools/src/browser/drivers/extension.ts WebSocket JSON-RPC 实现 BrowserDriver
浏览器插件 分别 ✗ apps/extension/ (独立项目) Chrome Extension — Service Worker + Content Script

六、Extension 通信协议

请求：Server → Extension 
{
  "id":      "req_a1b2c3",
  "method":  "navigate",
  "params": {
    "url": "https://example.com"
  }
}

{
  "id":      "req_d4e5f6",
  "method":  "click",
  "params": {
    "ref": "e14"
  }
}

{
  "id":      "req_g7h8i9",
  "method":  "snapshot",
  "params": {}
}
响应：Extension → Server 
{
  "id":      "req_a1b2c3",
  "result": {
    "url":   "https://example.com",
    "title": "Example Domain",
    "tree":  [/* SnapshotElement[] */],
    "text":  "page: Example Domain\n..."
  }
}

// 错误响应
{
  "id":    "req_d4e5f6",
  "error": {
    "code":    -32602,
    "message": "Element ref e14 not found"
  }
}

总结
共享（写一次）： 8 个 tool 定义 · BrowserDriver 接口 · snapshot 序列化 · 工厂入口 · 配置类型 分别实现： PlaywrightDriver（~200行）· ExtensionDriver（~150行）· 浏览器插件本体 已有代码改动： builtin-tools.ts 加一个 if 分支（~10行）
